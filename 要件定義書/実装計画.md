## 0. 前提

### 0.1 技術スタック確定事項

- **Frontend**: Next.js (App Router) + TypeScript + Tailwind CSS
- **State**: React Server Components + Zustand (クライアント側ゲーム状態)
- **Backend**: Firebase Cloud Functions (Node.js 20 / TypeScript)
- **DB**: Firestore (匿名認証ベース、ユーザー別ドキュメント)
- **LLM**: Firebase 経由の Gemini (無料枠) — Cloud Functions から `@google/generative-ai` または `@google-cloud/vertexai` で呼び出し
- **Hosting**: Firebase Hosting (Next.js は `firebase-frameworks` で連携)
- **Auth**: Firebase Anonymous Auth (将来 Google 連携可)
- **CI/CD**: GitHub Actions → Firebase Hosting & Functions

### 0.2 リポジトリ構成

```
2026AIHackathon/
├── apps/
│   └── web/                   # Next.js (Person B 主担当)
│       ├── app/
│       ├── components/
│       ├── lib/firebase/      # client SDK ラッパー
│       └── stores/            # Zustand stores
├── functions/                 # Cloud Functions (Person A 主担当)
│   ├── src/
│   │   ├── truthCompiler/     # Generator/Validator/Repairer
│   │   ├── runtimeSpeaker/    # 発言生成
│   │   ├── gameEngine/        # 状態遷移、勝敗判定
│   │   ├── llm/               # Gemini クライアント、JSON schema 検証
│   │   └── index.ts           # callable functions エクスポート
├── packages/

│       ├── types/             # CaseTruth, Character, Evidence, ...
│       ├── schemas/           # zod スキーマ (LLM出力検証 & Firestore検証)
│       └── contracts/         # Cloud Functions callable の型契約

├── firestore.indexes.json
├── firebase.json

```

> Monorepo は `pnpm workspaces` または `npm workspaces` で管理する。`packages/shared` は最初に切るのが最重要 (両者の境界を型で固定するため)。

### 0.3 並列作業の原則

1. **`packages/shared` を最初の半日で凍結**: ここが変わると両者がブロックされる。Phase 0 で合意して以降は PR レビュー必須。
2. **Cloud Functions の callable は契約ファースト**: Person A が実装より先に `contracts/` に signature を置く → Person B はスタブで先行可能。
3. **Firestore コレクションごとに owner を決める**: 同一ドキュメントへの同時書き込みを避ける。Phase 0 で割り当て。
4. **PR は 1 機能 1PR、main 直 push 禁止**。レビューは即日。

#### Firestore コレクション設計

```
users/{uid}
  └─ games/{gameId}
       ├─ meta              (current_day, current_phase, score, status)
       ├─ characters/{charId}     (public表示用のみ、secretは含めない)
       ├─ publicLogs/{logId}      (議論ログ)
       ├─ evidence/{evidenceId}   (プレイヤーが取得した証拠)
       ├─ interrogations/{intId}  (尋問履歴)
       ├─ pins/{pinId}            (ピン留め)
       └─ trials/{day}            (裁判結果)

internal/{gameId}             (★ Cloud Functions のみアクセス可)
  ├─ caseTruth                (固定真相全体、deduction_path含む)
  ├─ characterSecrets/{charId} (secret, private_goal, lie_policy, ...)
  └─ timeline/{eventId}       (夜間タイムライン)
```

> **重要**: `internal/` は Firestore security rules で `allow read, write: if false;` にし、Functions の Admin SDK 経由でのみアクセス。`users/{uid}/games/` 配下はオーナーのみ読み書き可。

## 3. 共有契約 (Phase 0 で確定する詳細)

### 3.1 主要型 (`packages/shared/types/`)

要件定義書 §12 をそのまま TypeScript に: `CaseTruth`, `Character`, `TimelineEvent`, `Evidence`, `Testimony`, `DialogueLog`, `InterrogationAction`, `DeductionPath`, `TrialDecision` を必須。

加えて MVP 固有:

```ts
type GameMeta = {
  gameId: string;
  uid: string;
  currentDay: 1 | 2 | 3;
  currentPhase:
    | 'morning'
    | 'discussion'
    | 'investigation'
    | 'organize'
    | 'trial'
    | 'night'
    | 'result';
  remainingPoints: number;
  aliveCharacters: string[];
  villageTrust: number; // 0-100
  status: 'in_progress' | 'won' | 'lost';
  createdAt: Timestamp;
};
```

### 3.2 LLM 出力スキーマ (`packages/shared/schemas/`)

zod で全 LLM 出力を schema 化。Generator/Validator/Speaker それぞれ別ファイル。

### 3.3 Firestore セキュリティルール骨子

```
match /users/{uid}/games/{gameId}/{document=**} {
  allow read: if request.auth.uid == uid;
  allow write: if false; // クライアントから直接書かない、全部 Functions 経由
}
match /internal/{document=**} {
  allow read, write: if false;
}
```

### 3.4 環境変数

- `GEMINI_API_KEY` (Functions のみ、Secret Manager 経由)
- `NEXT_PUBLIC_FIREBASE_*` (Web 用 config)
- `NEXT_PUBLIC_USE_MOCK` (開発時のスタブ切替)

## 4. リスクと対策

| リスク                                     | 影響                      | 対策                                                                                          |
| ------------------------------------------ | ------------------------- | --------------------------------------------------------------------------------------------- |
| Truth Compiler の検証通過率が低い          | ゲーム開始が遅い/失敗する | 段階生成 + Repairer 最大 3 回 + 失敗時はテンプレ骨格にフォールバック                          |
| Gemini 無料枠のレート制限                  | デモ時に詰まる            | 1 ゲームあたりの LLM 呼び出し回数を計測、キャッシュ可能な部分はキャッシュ、デモ用シードを用意 |
| LLM 生成 JSON のスキーマ不一致             | 発言が出ない              | zod 検証 → 自動リトライ最大 3 回 → fallback テキスト                                          |
| 知識範囲外の発言 (要件 §4.4)               | 推理が破綻                | プロンプトで明示 + 出力に `known_facts_used` を含めさせて Functions 側で照合                  |
| 並列作業のマージ競合                       | 進捗ロス                  | `packages/shared` 以外は基本独立、共有部分は PR レビュー必須                                  |
| 尋問の自由入力                             | スコープ膨張              | MVP は選択式のみ (要件 §10.6 の方針通り)、Could に分類                                        |
| Next.js + Firebase Hosting の SSR 連携の癖 | デプロイで詰まる          | Phase 1 で `firebase deploy` の試運転を 1 回通しておく                                        |
